In an on-demand platform like your car wash app, using a traditional relational database (MySQL) for absolutely everything will quickly lead to performance bottlenecks.

In an interview, if you are asked why you chose Redis alongside MySQL, your core argument should be **speed and specialized data structures**. MySQL is disk-based and great for permanent, structured records (like payments and wash history), but Redis is an in-memory datastore designed for lightning-fast, real-time operations.

Here are the three specific reasons your system needs Redis:

### 1. High-Frequency Live Location Tracking (Geospatial Data)

When a washer is "Online," their mobile app will ping the backend with their current GPS coordinates (latitude/longitude) every 3 to 5 seconds.

* **The MySQL Problem:** Running an `UPDATE` query on a MySQL table every 3 seconds for thousands of active washers will overwhelm the database with heavy write locks.
* **The Redis Solution:** Redis handles millions of read/write operations per second. More importantly, Redis has built-in **Geospatial commands** (`GEOADD`, `GEORADIUS`). When a user requests a wash, you can ask Redis, *"Give me all online washers within a 5km radius of the user's location,"* and it will return the list in sub-milliseconds.

### 2. Distributed Locking (Concurrency Control)

We discussed the "Uber logic" where a request is broadcasted to multiple washers.

* **The Problem:** Two washers might tap the "Accept" button at the exact same millisecond. If the system processes both, two washers will be assigned to one car.
* **The Redis Solution:** You can use Redis to implement a **Distributed Lock** using atomic commands (like `SETNX` - Set if Not eXists). The first request to hit Redis locks the wash job ID. The second request hits Redis a millisecond later, sees the lock is already there, and gets rejected.

### 3. Ephemeral Data and TTL (Time-To-Live)

You only care about a washer's location *while they are online*.

* **The Problem:** If a washer's phone battery dies suddenly, they won't send an "Offline" signal. MySQL would still think they are online and assign them jobs.
* **The Redis Solution:** Redis allows you to set a TTL (Time-To-Live) on data. You can set a washer's location data to expire every 15 seconds. If their app stops sending the 5-second pings, Redis automatically deletes them from the "Active Washers" pool, preventing dead assignments.

| --- | --- | --- |
| **Best For** | Permanent data, complex relations, financial records. | Temporary data, high-speed tracking, caching. |
| **Used For (Your App)** | User profiles, Wash History, Payments. | Active Washer tracking, Real-time matching, Locks. |
| **Speed** | Milliseconds. | Sub-milliseconds / Microseconds. |

## 🚗 High-Level Design (HLD): On-Demand Car Wash Platform

### 1. System Overview & Actors

The platform connects users needing a car wash with nearby available washers in real-time. It operates on a service-oriented architecture ensuring seamless booking, tracking, and payment.

* **Customer (User):** Requests services, tracks washers, and makes payments.
* **Washer (Service Provider):** Receives requests, manages availability, and fulfills services.
* **Admin:** Manages user/washer accounts, monitors platform health, and handles disputes.
* **External Systems:** Payment Gateway (Stripe/Razorpay) and Notification Service (FCM/Twilio).

---

### 2. Architecture Overview

The system relies on a robust tech stack to handle real-time location data and transactional integrity.

* **Frontend:** React (Web/Mobile App)
* **Backend:** Java Spring Boot
* **Database:** MySQL (Relational data)
* **Caching/In-Memory:** Redis (Active washer tracking & matching)

### 3. Core Microservices / Components

| Service | Primary Responsibilities |
| --- | --- |
| **User Service** | Auth, profile management, car details, and payment initiation. |
| **Washer Service** | Auth, online/offline toggling, and live location broadcasting. |
| **Request Service** | State machine for wash requests (`REQUESTED` → `ACCEPTED` → `IN_PROGRESS` → `COMPLETED` → `CANCELLED`). |
| **Matching Service** | Radius-based searching, request broadcasting, and handling the "First Accept" logic. |
| **Payment Service** | Third-party integration, transaction logging, and status updates. |
| **Notification Service** | Async delivery of Push/SMS alerts across different request states. |

---

### 4. The Matching Service (Core Logic)

The Matching Service is the heart of this platform. The logic of finding nearby washers, sending requests, and locking the assignment to the first responder is highly comparable to an ambulance-based system where real-time location tracking and immediate, precise dispatch are critical.

**Concurrency Control (Interview Gold):**
To prevent two washers from accepting the exact same wash request simultaneously, the system must implement strict locking.

* **Solution:** Implement Optimistic Locking (using a `@Version` annotation in Java Spring Boot) on the MySQL database row, or utilize a Redis Distributed Lock for faster, in-memory resolution.

---

### 5. Database Schema (Conceptual)

| Table Name | Key Columns | Description |
| --- | --- | --- |
| **Users** | `id`, `name`, `phone`, `role` | Stores all platform identities (USER/WASHER/ADMIN). |
| **Washer_Profile** | `id`, `user_id`, `is_online`, `lat`, `long` | Tracks real-time availability and location. |
| **Wash_Request** | `id`, `user_id`, `washer_id`, `status`, `price` | The central transactional record of the service. |
| **Payment** | `id`, `request_id`, `amount`, `status` | Ledger for all financial transactions. |

---

### 6. The Request Flow Lifecycle

1. **Request Initiation:** User clicks "Book Wash". A record is created with status `REQUESTED`.
2. **Broadcast:** The Matching Service queries Redis for online washers within a 5km radius and fires notifications.
3. **Lock & Assign:** The first washer to hit "Accept" triggers a lock on the request. The `washer_id` is assigned, and the status updates to `ACCEPTED`. Other pending notifications expire.
4. **Execution:** Washer arrives and updates status to `IN_PROGRESS`.
5. **Completion & Payment:** Washer marks `COMPLETED`. User is prompted to pay, changing the status to `PAID`. Washer is marked `is_online = true` again.

---

### 7. Scalability & Future-Proofing

As the platform grows, the architecture can evolve:

* **Load Balancing:** Distribute incoming API requests.
* **Event-Driven Processing:** Introduce Kafka or RabbitMQ to decouple the Notification and Payment services from the main request thread.
* **WebSockets:** Shift from HTTP polling to WebSockets for live, map-based tracking of the washer.
